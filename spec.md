### **Project Specification: Automated Email Processing and AI-Powered Reply Bot**

#### **1. High-Level Description**

This document outlines the requirements for a Python script that automates email handling. The script will monitor a local directory for new email files, use a Large Language Model (LLM) via either the Ollama API or the llama.cpp HTTP API to understand the email's content, and send an intelligent reply.

#### **2. Core Workflow & Logic**

The script operates in a continuous, single-threaded polling loop.

1.  **Monitor:** The script checks a specified `source_folder` for the presence of any email files every `polling_interval_seconds`. This is a Maildir-type email folder, so all files in the directory will contain exactly one email.
2.  **Process:** It processes files one at a time.
3.  **Extract Data:** For each email file, it performs the following extractions:
    *   **Headers:** The sender's email address (`From`), the `Subject`, `Cc`, and the message id headers: `Message-Id`, `In-Reply-To` and `References`.
    *   **Body:** It extracts the email body content. It MUST prioritize the `text/plain` part of the email. If a plain text part is not available, it MUST fall back to using the `text/html` part.
4.  **Reformat Reply Chains:** If the email body contains a reply separator line matching the pattern `On <date> <sender> wrote:`, the body is reformatted into chronological chat order before further processing (see Section 6).
5.  **Process Attachments:**
    *   It inspects all attachments and compares their file extensions against a configurable `allowed_attachment_extensions` list (e.g., `['.txt', '.md', '.eml']`).
    *   If any attachment has a disallowed extension, the script triggers the "Unsupported Attachment Error" flow (see Section 3).
    *   For all allowed attachments, it reads their text content.
6.  **Construct API Prompt:** It constructs a single text prompt for the LLM API in the following order and format:
    1.  The content of `ollama_prefix_prompt` or `llamacpp_prefix_prompt` from the configuration (depending on the active backend).
    2.  The text content of each allowed attachment, wrapped in XML-style tags: `<attachment filename="[original_filename]">[attachment_text_content]</attachment>`.
    3.  The (possibly reformatted) email body, wrapped in an XML-style tag: `<email_body>[email_body_content]</email_body>`.
    *   If the email body is empty but attachments are present, the `<email_body>` tag should be omitted.
7.  **Interact with LLM API:** It sends the complete prompt to the configured LLM backend (`ollama` or `llamacpp`), selected via the `llm_backend` config key.
8.  **Compose Reply:**
    *   **Subject:** The reply subject line MUST be prefixed with `Re: `, unless the original subject already starts with `Re: `.
    *   **Body:** The reply email body MUST be structured as follows:
        1.  The full (possibly reformatted) email body — the complete conversation history, oldest message first.
        2.  A blank line.
        3.  A line containing: `Assistant:`
        4.  The text response generated by the LLM.
        5.  A blank line.
        6.  A line containing: `[Place any reply down here at the bottom.]`
9.  **Send Reply:** The composed reply is sent via SMTP to sender and anyone on the CC header.
10. **Archive:** Upon successful sending, the original email file is moved from the `source_folder` to the `processed_folder`.

#### **3. Error Handling**

*   **API Failure:**
    *   **Retry:** If a call to the LLM API fails, the script MUST retry the call up to 3 times.
    *   **Notify & Archive:** If all 3 retries fail, the script MUST send an email to the original sender with the error details, then move the email file to the `processed_folder`.
*   **SMTP Failure:**
    *   **Log & Halt:** If the script fails to send an email via SMTP, it MUST log the error.
    *   **Do Not Archive:** It MUST NOT move the email file, ensuring the email is attempted again on the next run.
    *   **Terminate:** The script should exit after an SMTP failure.
*   **Unsupported Attachment Error:**
    *   **Notify:** The script MUST send a reply to the original sender stating which attachments were rejected and listing the allowed types.
    *   **Archive:** After sending the notification, the script MUST move the email file to the `processed_folder` and continue processing other files.

#### **4. Logging**

*   **Level:** Log events at the `INFO` level of detail.
*   **Destination:** Logs should be printed directly to the console.
*   **Content:** Logs should include messages for key actions, such as starting to process a file, successful replies, and all error conditions.

#### **6. Reply Chain Reformatting**

When the bot receives a reply to one of its own emails, the user's new message appears at the top and the prior conversation is quoted below a separator line of the form:

```
On Wed, Mar 4, 2026, 15:45 <bot@example.com> wrote:
```

The script reformats this into chronological (chat-style) order before passing it to the LLM and including it in the outgoing reply:

1.  **Detect separator:** Find the `On ... wrote:` line. This line may be split across two consecutive lines (some email clients wrap long headers); both forms are handled.
2.  **Split:** Divide the body into `user_reply` (text before the separator) and `quoted_body` (text after the separator). The separator line itself is discarded.
3.  **Strip quotes:** Remove all leading `>` (and `> `) characters from every line of `quoted_body`.
4.  **Reflow:** Join consecutive non-blank lines in `quoted_body` into a single long line. Blank lines are preserved as paragraph separators. This undoes line-wrapping introduced by email clients.
5.  **Reassemble:** Return `quoted_body` followed by a blank line and then `user_reply`, placing the oldest content at the top and the newest at the bottom.

If no separator is found (i.e., the email is not a reply), the body is passed through unchanged.
